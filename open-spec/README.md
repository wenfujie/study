# openSpec上手

## 1. OpenSpec 是什么？

OpenSpec 是一个**规范驱动开发**工具，专为 AI 编码助手设计。核心就一句话：**写代码之前，先让人类和 AI 把“要做什么”定下来**。

解决ai coding痛点：需求散落在聊天记录里，AI 容易遗漏需求、偏离预期。OpenSpec 把模糊的提示变成**可审查、可落地的工程计划**，从此告别“看着像对、跑起来全错”的随机性。

---

## 2. 安装与初始化

**环境要求**：Node.js 20.19.0 或更高版本

```bash
# 全局安装
npm install -g @fission-ai/openspec@latest

# 进入项目目录并初始化
cd your-project
openspec init
```

初始化会在项目根目录创建 `openspec/` 文件夹结构和配置文件。

---

## 3. 目录结构

初始化后的 `openspec/` 目录长这样：

```
openspec/
├── specs/                 # 已实现的功能（唯一真相源）
│   └── [domain]/
│       └── spec.md
├── changes/               # 待实现的变更
│   ├── [change-name]/
│   │   ├── proposal.md    # 为什么做 + 做什么
│   │   ├── design.md      # 技术方案
│   │   ├── tasks.md       # 实施清单（可勾选）
│   │   └── specs/         # 规范增量
│   └── archive/           # 归档（已完成变更）
```

**核心理念**：`specs/` 是当前系统的“真相”，`changes/` 是正在进行的变更，每个变更独立成文件夹，归档后合并回主规范。

---

## 4. 核心工作流

OpenSpec 提供两种工作流模式，新手从默认快速模式开始即可。

### 默认快速模式

适用于需求明确、流程简单的场景。

| 命令                   | 作用                                                          | 使用时机             |
| ---------------------- | ------------------------------------------------------------- | -------------------- |
| `/opsx:propose [描述]` | 创建变更 + 生成所有规划文档（proposal、specs、design、tasks） | 新功能/需求明确时    |
| `/opsx:explore [问题]` | 纯思考模式，AI 只分析不写代码                                 | 需求澄清、技术调研   |
| `/opsx:apply`          | 按 tasks.md 逐条实现代码                                      | 规划文档准备好后     |
| `/opsx:archive`        | 归档已完成的变更，更新主规范                                  | 所有任务完成并验证后 |

> 💡 `/opsx:propose` 是“一把梭”命令，一条命令把提案、规范、设计、任务全生成好，适合明确的需求。

### 扩展工作流

适用于复杂开发、团队协作场景。通过 `openspec config profile` 开启后，可使用更多命令：

| 命令            | 作用                         |
| --------------- | ---------------------------- |
| `/opsx:new`     | 创建变更基础脚手架           |
| `/opsx:ff`      | “快进”——一键生成所有规划文档 |
| `/opsx:verify`  | 验证代码与规范的一致性       |
| `/opsx:sync`    | 将增量规范合并到主规范       |
| `/opsx:onboard` | 交互式教程                   |

## 5. 常用 CLI 命令

| 命令                                 | 用途                               |
| ------------------------------------ | ---------------------------------- |
| `openspec list`                      | 查看当前所有进行中的变更及任务进度 |
| `openspec show <change-id>`          | 查看某个变更的详细信息             |
| `openspec validate <change-id>`      | 检查规范格式是否符合标准           |
| `openspec archive <change-id> --yes` | 归档变更（非交互模式）             |
| `openspec update`                    | 升级后更新配置和 AI 文档           |

> 💡 大部分日常操作由 AI 通过斜杠命令完成，CLI 主要用于查看进度、验证格式等管理操作。

---

## 6. 实操示例

以“添加深色模式”为例，演示完整流程：

```bash
# 1. 创建变更（AI对话中操作）
/opsx:propose add-dark-mode
# AI自动创建 openspec/changes/add-dark-mode/ 并生成规划文档

# 2. 审查并确认规划文档（手动检查 proposal.md / design.md / tasks.md）

# 3. 开始实施
/opsx:apply
# AI 逐步完成任务清单中的任务

# 4. 所有任务完成后，归档
/opsx:archive
# AI 将变更归档并更新主规范
```

---

## 7. 核心原则

| 原则           | 含义                                                         |
| -------------- | ------------------------------------------------------------ |
| **棕地优先**   | 不要求从零开始，与现有项目完美兼容                           |
| **规范即真相** | `specs/` 是已实现功能的唯一真相源，`changes/` 是待实施的提案 |
| **增量变更**   | 每个变更独立成文件夹，归档后合并，历史清晰可追溯             |
| **人在回路**   | AI 生成提案后必须经人工审查确认，再进入实施阶段              |

---

## 8. 总结

| 步骤       | 操作                                  |
| ---------- | ------------------------------------- |
| ① 安装     | `npm install -g @fission-ai/openspec` |
| ② 初始化   | `openspec init`                       |
| ③ 创建变更 | `/opsx:propose <你的想法>`            |
| ④ 审查文档 | 检查 proposal/specs/design/tasks      |
| ⑤ 实施     | `/opsx:apply`                         |
| ⑥ 归档     | `/opsx:archive`                       |

核心就是一**规**到底：**规范先行，AI 执行，过程可追溯**。不需要把整个项目背下来，OpenSpec 帮你把“要做什么”写在文件里，AI 照着规范干活，你再也不用担心它跑偏了。
