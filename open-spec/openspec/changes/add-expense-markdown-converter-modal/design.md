## Context

花销统计页已经具备表格录入与实时合计能力，但仍缺少把表格快速转成可复制 Markdown 的能力。当前页面还没有通用弹窗抽象，因此如果直接把转换弹窗写死在页面里，会导致后续页面复用成本更高。

## Goals / Non-Goals

**Goals:**
- 为花销统计页提供“markdown表格转换”入口。
- 将当前表格内容转换为 Markdown 表格语法并默认回填到 `textarea`。
- 支持一键复制到剪切板和关闭后清空内容。
- 抽象一个可复用的通用弹窗组件。

**Non-Goals:**
- 不支持真正富文本编辑能力；输入控件采用 `textarea`。
- 不处理复杂表格样式或多格式导出。
- 不引入第三方剪切板或弹窗依赖。

## Decisions

1. 通用弹窗采用插槽 + 显式关闭控制。
- Rationale: 弹窗内容需求简单，但后续复用空间明确；插槽可以让内容区保持灵活，而关闭事件由父组件统一控制，状态更清晰。
- Alternatives considered: 每个页面单独写弹窗；实现快但不可复用且难维护。

2. Markdown 转换逻辑放在页面侧的纯函数中。
- Rationale: 转换依赖当前表格数据，属于业务逻辑，放在可测试的纯函数里更容易验证；弹窗组件只负责展示，不耦合数据结构。
- Alternatives considered: 直接写在弹窗内部；会把 UI 与业务转换强绑定，降低复用性。

3. 剪切板复制优先使用 `navigator.clipboard.writeText`。
- Rationale: 原生 API 简洁且符合现代浏览器支持；在失败时可以通过提示或降级流程保留可用性。
- Alternatives considered: 使用隐藏文本域和 `execCommand('copy')`；兼容性更广但实现更旧且更脆弱。

4. 关闭弹窗时重置编辑内容为空。
- Rationale: 需求明确要求关闭即清空，这能避免下一次打开时残留旧内容，减少误复制风险。
- Alternatives considered: 保留上次内容；不符合明确交互要求。

## Risks / Trade-offs

- [风险] 剪切板 API 在非安全上下文或权限受限时可能失败 → Mitigation: 保留失败反馈并避免破坏页面状态。
- [风险] Markdown 转换格式若未统一会影响可复制性 → Mitigation: 将转换函数固定为确定性输出并为其添加测试。
- [权衡] 通用弹窗增加一点抽象层，但可显著提升后续复用能力 → Mitigation: 只抽象最小必要能力（标题、默认 slot、关闭事件）。

## Migration Plan

- 直接在现有花销统计页中替换或补充交互，不需要数据迁移。
- 新增通用弹窗组件后，先只被当前页面消费；后续页面可按需复用。
- 如果复制流程出现问题，可临时保留弹窗与 Markdown 文本展示，但不影响主表格录入功能。
