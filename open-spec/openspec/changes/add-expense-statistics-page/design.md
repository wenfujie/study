## Context

当前项目需要新增一个前端页面用于花销录入与金额汇总。该页面以表格形式展示并编辑数据，每行包含“花销项”和“花销金额”两个字段。金额输入需要具备格式约束（仅数字，精度 2 位），并在用户编辑时实时计算总金额 `amount`，以保证反馈及时且结果可见。

## Goals / Non-Goals

**Goals:**
- 提供包含两列输入框（花销项、花销金额）的可编辑表格。
- 对金额输入进行前端约束：仅允许数值语义，最多两位小数。
- 在页面下方展示实时动态计算的总金额 `amount`（金额列合计）。
- 保持实现简单，可在当前前端架构中直接接入。

**Non-Goals:**
- 不包含后端持久化与接口对接。
- 不包含复杂财务规则（币种换算、税率、分摊规则）。
- 不包含导出、导入或历史版本管理。

## Decisions

1. 使用受控输入与本地状态管理表格数据。
- Rationale: 受控输入可在每次变更时同步触发校验与合计计算，行为可预测，便于测试。
- Alternative considered: 非受控输入 + 提交时统一解析；该方案实时反馈较弱，不满足“实时动态计算”要求。

2. 金额输入采用“输入阶段容忍 + 计算阶段规范化”的策略。
- Rationale: 用户输入过程中可能出现中间态（如 `12.`），若强制即时格式化会影响输入体验；在计算时将可解析值纳入合计并统一保留两位小数展示。
- Alternative considered: 每次按键后强制格式化为两位小数；该方案输入体验较差，光标与编辑行为容易受干扰。

3. 总金额以派生值计算，不单独存储。
- Rationale: `amount` 完全由金额列推导，作为派生数据可避免状态冗余与不一致。
- Alternative considered: 额外维护总金额状态；会带来同步风险与额外复杂度。

## Risks / Trade-offs

- [风险] 金额输入边界（空值、前导零、小数点中间态）处理不一致 → Mitigation: 统一解析函数并覆盖常见边界场景。
- [风险] 频繁输入触发重算影响性能（大数据量） → Mitigation: 当前页面规模较小，先采用直接求和；若行数增大再引入优化。
- [权衡] 容忍输入中间态提升体验，但校验逻辑更复杂 → Mitigation: 将“可输入格式”与“可计算格式”职责分离。
